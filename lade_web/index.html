Design Motivation

Why last-mile delivery deserves a purpose-built database

Last-mile delivery is the most expensive and time-consuming stage of e-commerce logistics. A major platform handles millions of orders daily, coordinates tens of thousands of couriers, and dispatches dynamically across thousands of AOIs (Areas of Interest).

The raw LaDe dataset stores orders as a wide table with heavy redundancy: every order row repeats the courier's home city, the AOI's region, and its type. At production scale this redundancy dramatically inflates storage cost and update latency.

We decomposed the wide table into 6 relations satisfying 3NF, eliminating transitive dependencies and separating fact tables (orders, trajectories) from dimension tables (couriers, AOI, road network). Composite indexes were added for high-frequency query patterns so the system can serve both operational workloads and analytical data mining.

Index Decisions

Step 8 · Index strategy derived from observed query patterns

Courier_Trajectories (courier_id, gps_time)

Composite index. Most trajectory queries ask "show courier X's track between times A–B". A composite index on these two columns enables tight range scans and is by far the most impactful.

Pickup/Delivery_Orders (courier_id)

Speeds up JOINs for the frequent "show this courier's tasks today" pattern and avoids full-table scans.

Pickup/Delivery_Orders (aoi_id)

Supports AOI-level heatmaps and area-wise aggregations used in analytics.

Pickup/Delivery_Orders (ds)

Business queries almost always filter by date; indexing ds drastically cuts scan volume.

AOI_Master (city, region_id)

Enables hierarchical filtering, e.g. "list all AOIs in region R of city C".

Couriers · PRIMARY KEY

SQLite auto-creates a B-tree on the primary key, giving O(log n) courier lookups for free.

Implementation Notes

Demo Environment Disclosure
Transparency

This demo uses SQLite running entirely in the browser so that graders can try it with zero setup. Our production schema (Section 4 of the report) is written in MySQL dialect; minor differences in DDL and date functions have been accommodated in the preset queries.

Date normalization: the raw CSV stores timestamps as MM-DD HH:MM:SS (no year). During ETL we prepend 2024- so that julianday() and other time functions work correctly.

Sample size: for browser performance we load 10,000 pickup + 10,000 delivery + 1,000 trajectory records from Jilin. The schema scales unchanged to the 10,677k-package scale reported in the original paper.

Normalization Rationale

3NF · Eliminating Transitive Dependencies

In the raw wide table, aoi_id → city, region_id forms a transitive dependency. After extracting AOI_Master, each order row stores only aoi_id; changing a city propagates by updating a single row.

Separating Facts from Dimensions

Fact tables (orders, trajectories) grow continuously; dimension tables (couriers, AOI) are small and slowly-changing. Separating them lets us apply different indexing and caching strategies.

Pickup vs Delivery as Separate Tables

Pickup orders have time_window; delivery orders do not. Merging would introduce many NULLs. Separate tables keep each schema complete and allow NOT NULL constraints.

Trajectories as a Standalone Table

GPS trajectories arrive every ~20 seconds — high-frequency time-series data. Attaching them to orders would create multivalued dependencies. A dedicated table supports efficient (courier_id, gps_time) indexing.

Architecture & Prompt Design

Architecture: the browser POSTs the user's question to a lightweight Flask backend running on localhost:5001, which forwards it to DeepSeek V3.2 together with our schema prompt. The returned SQL is executed against the SQLite database by sql.js in the browser. A backend proxy is required because the DeepSeek API key must never be exposed in client-side JavaScript, and because browsers cannot directly call external LLM APIs due to CORS restrictions.

Fault tolerance: if the backend is unreachable — during an offline demo, or when the network is down — the frontend transparently falls back to a curated cache of 8 representative question/SQL pairs. The status indicator clearly marks whether each answer came from a live LLM call or from the cached fallback. This two-layer design guarantees the demo works in every scenario.

System prompt sent to DeepSeek consists of three critical elements:

① Schema context — all six tables with fields, primary keys, and foreign keys.

② Dialect constraint — explicitly require SQLite syntax (e.g. julianday() instead of TIMESTAMPDIFF).

③ Output format constraint — return raw SQL only, no markdown or commentary, so it can be parsed and executed directly.

④ Safety guardrail — the backend rejects any generated SQL containing INSERT / UPDATE / DELETE / DROP, ensuring read-only access.

The full prompt text and a comparison study of 10 well-formed vs. ablated prompts appears in the report appendix.
